# Завдання 2 - Семантичний пошук за науковими статтями

Навчальний проєкт курсу **«NoSQL та векторні бази даних» (GoIT)**.
Будуємо повний пайплайн від «сирих» текстів наукових статей до робочого
семантичного пошуку: ембединги → векторний індекс → фільтрація за метаданими →
гібридний пошук (BM25 + вектори) із злиттям через **RRF**.

---

## Опис проєкту

Пошуковий рушій для наукових статей. Користувач ставить запитання звичайною
мовою (наприклад, *«graph neural networks for molecules»*), а система знаходить
релевантні статті **за змістом**, навіть якщо в них вжито інші слова, ніж у
запиті (семантика, а не збіг токенів).

Датасет - підмножина **arXiv** (абстракти статей із полями `title`, `authors`,
`year`, `category`). Модель ембедингів - **SPECTER2** (`allenai/specter2_base`),
спеціально натренована на наукових документах та графі цитувань.

Проєкт повністю запускається **офлайн** завдяки двом резервним механізмам:

- синтетичний міні-датасет (`data/sample.jsonl`, ~40 записів) замість великого
  файлу з Kaggle;
- локальний in-memory косинусний індекс (`USE_LOCAL=1`) замість Pinecone;
- автоматичний відкат на меншу модель `all-MiniLM-L6-v2`, якщо SPECTER2 не
  завантажується.

---

## Архітектура пайплайну

Шість скриптів виконуються **послідовно**, кожен спирається на результат
попереднього.

| Крок | Скрипт | Що відбувається | Вхід → Вихід |
|------|--------|-----------------|--------------|
| 1 | `scripts/01_prepare_data.py` | Читає arXiv JSONL, чистить (whitespace, дублі, короткі абстракти), бере репрезентативну **reservoir-вибірку** 5k–10k по всьому дампу (спан за роками), фіксує схему `id, title, abstract, authors, year, category`. | `data/sample.jsonl` (або Kaggle dump) → `data/arxiv_subset.parquet` |
| 2 | `scripts/02_embed.py` | Кодує `title [SEP] abstract` моделлю SPECTER2, робить **L2-нормалізацію**, зберігає вектори та мапінги. | `arxiv_subset.parquet` → `embeddings/embeddings.npy`, `id_map.json`, `metadata.json`, `model_info.json` |
| 3 | `scripts/03_load_to_pinecone.py` | Створює Pinecone-індекс (метрика **cosine**, правильна розмірність) і робить **batch-upsert** векторів із метаданими. У режимі `USE_LOCAL=1` - будує/перевіряє локальний індекс. | `embeddings/*` → Pinecone або in-memory індекс |
| 4 | `scripts/04_search.py` | Три режими пошуку: **(a)** чистий семантичний; **(b)** семантичний + фільтри за `year`/`category`; **(c)** порівняння метрик (cosine / dot / euclidean). | запит → ранжований список статей |
| 5 | `scripts/05_chunking.py` | Бере **30 статей із найдовшими абстрактами**, ріже їх двома стратегіями (**fixed** з невеликим overlap та **sentence-based**), будує **окремий індекс на стратегію** (локальний namespace або окремий Pinecone-індекс), ембедить чанки з метаданими й batch-upsert, потім друкує top-5 чанків на стратегію для тестових запитів. | `arxiv_subset.parquet` → два чанк-індекси + результати пошуку |
| 6 | `scripts/06_hybrid_search.py` | Лексичний **BM25** + векторний пошук, злиття рангів через **RRF (k=60)**. Показує, що гібрид перевершує кожен метод окремо. | запит → BM25 / vector / hybrid ранжування |

Допоміжний модуль `scripts/common.py` тримає спільні шляхи, завантажувач моделі
(SPECTER2 з відкатом) та клас `LocalCosineIndex` - реальний робочий локальний
векторний індекс.

## Датасет і як його отримати

Повний датасет - **«arXiv Dataset»** від Cornell University на Kaggle:

> https://www.kaggle.com/datasets/Cornell-University/arxiv

Це один великий JSONL-файл (`arxiv-metadata-oai-snapshot.json`, ~4 ГБ). Через

```bash

    --input data/arxiv-metadata-oai-snapshot.json \
```

прогнати офлайн. Параметр `--max-records` тримайте в діапазоні 5000–10000 для
реального датасету.

```bash
python scripts/05_chunking.py --recreate

# 6. Гібридний пошук BM25 + вектори через RRF (3 тестові запити за замовчуванням)
python scripts/06_hybrid_search.py
```

## Відповіді на питання ДЗ

### Частина 1.2, питання 1: чим Pinecone відрізняється від Qdrant і Chroma

Порівняння за моделлю розгортання, ліцензією та продуктивністю:

| | Pinecone | Qdrant | Chroma |
|---|---|---|---|
| Модель розгортання | повністю керований **SaaS** (serverless у хмарі провайдера) | **open-source**: self-host (Docker/k8s) або керована Qdrant Cloud | легка **вбудована**/локальна БД (in-process), є й серверний режим |
| Ліцензія | **пропрієтарна**, закрита | відкрита (**Apache 2.0**), ядро на **Rust** | відкрита (**Apache 2.0**), Python/TS |
| Продуктивність / масштаб | дуже добре масштабується горизонтально, ANN-індекс і інфраструктуру тримає провайдер; платиш за керованість | висока швидкість завдяки Rust і HNSW; повний контроль над тюнінгом, фільтрами, шардингом | оптимізована під простоту й малі/середні обсяги; не для білйон-масштабу з коробки |

**Коли що обирати:**
- **Pinecone** - коли потрібен production-сервіс без обслуговування інфраструктури
  (керований SaaS), і команда готова платити за зручність та масштабування.
- **Qdrant** - коли потрібен **контроль і self-host** (on-prem, приватні дані,
  тонке налаштування), або керована хмара з відкритим ядром; добре тримає
  складні фільтри за метаданими.
- **Chroma** - коли потрібен **швидкий прототип / локальна розробка** і
  вбудована БД у тому ж процесі (RAG-демо, ноутбуки), без окремого сервісу.

### Частина 1.2, питання 2: чому SPECTER2, а не універсальна модель (all-MiniLM-L6-v2)

`all-MiniLM-L6-v2` - **універсальний** sentence-енкодер загального призначення:
він добре працює на «звичайних» реченнях, але не знає специфіки наукового тексту.
**SPECTER2** (`allenai/specter2_base`) натренована саме на **наукових статтях** і
на **графі цитувань**: пари статей, що цитують одна одну, притягуються в просторі
ембедингів. У картці моделі на HuggingFace зазначено, що вона дає **document-level
representation наукових документів** і призначена для задач на кшталт
**retrieval, classification та citation prediction**. Тобто для пошуку за
науковими абстрактами SPECTER2 кодує саме ту «наукову» семантику й споріднення за
цитуваннями, яку універсальна модель усереднює, тому вона тут точніша.

### Частина 1.2, питання 3: яку метрику схожості рекомендує картка моделі і чому це важливо для індексу

Картка моделі SPECTER2 рекомендує **косинусну подібність (cosine)**. Це важливо,
бо **метрика індексу має збігатися з тим, як модель навчалася й нормалізувалася**:
SPECTER2 оптимізована контрастивно під косинус, тому в `02_embed.py` ми робимо
**L2-нормалізацію**, а індекс у `03_load_to_pinecone.py` створюємо з
`metric=cosine`. Якщо взяти «чужу» метрику, ранжування або значення відстаней не
відповідатимуть геометрії, у якій модель розрізняє документи, і якість пошуку
впаде.

### Питання до етапу ембедингів: чому для L2-нормалізованих векторів cosine = dot product

Косинусна подібність за визначенням:

```
cos(a, b) = (a · b) / (|a| · |b|)
```

Після L2-нормалізації кожен вектор має одиничну довжину: `|a| = |b| = 1`. Тоді
знаменник дорівнює 1, і формула спрощується до

```
cos(a, b) = a · b
```

Тобто для **unit-length** ембедингів косинус **тотожно дорівнює скалярному
добутку**. Саме тому локальний індекс (`LocalCosineIndex`) зберігає нормалізовані
вектори й рахує косинус простим dot product, а cosine та dotproduct у режимі
`--mode metric` дають однакові бали.

### Питання до етапу chunking: яка стратегія дає змістовніші чанки

**Sentence-based** зазвичай дає **змістовніші** чанки: межі проходять по реченнях,
тож кожен чанк - самодостатній смисловий блок. У **fixed**-стратегії межа падає
**посеред речення**, тому думку може бути розрізано навпіл: половина потрапляє в
один вектор, половина - в інший, і обидва ембединги стають «розмитими» та гірше
відповідають на запит, якому потрібне ціле речення (невеликий overlap лише
частково рятує). Розмір чанка теж впливає прямо: **замалий** чанк (10-15 слів)
не має контексту, і ембединг шумить; **завеликий** повертає нас до усереднення
деталей. Оптимум **залежить від задачі** й має узгоджуватися з контекстним вікном
моделі та стилем запитів.

На реальному прогоні `05_chunking.py` це видно прямо: у **fixed**-стратегії
топовими чанками ставали обрізки на кшталт «that are optimal up to an additive
term of 1.» (score 0.95) або «produced constitutes a minimum spanning tree of the
initial graph.» (score 0.96) - висока косинусна оцінка, але фрагмент вирваний із
середини речення, без контексту. **Sentence**-стратегія для тих самих статей
повертала цілі, самодостатні речення. Це і є практичний доказ, що нарізання
посеред речення псує інтерпретованість чанка (хоча на коротких абстрактах обидві
стратегії дають близькі за релевантністю результати, бо чанків мало).

## Висновки

### 1. Порівняння методів 3x3 (BM25 / vector / hybrid)

`06_hybrid_search.py` ганяє **3 запити** через **3 методи**. Нижче -
**реальні** результати прогону на представницькій вибірці (8000 статей):

| Запит | BM25 | Vector | Перетин top-5 | RRF відновив |
|---|---|---|---|---|
| `reciprocal rank fusion for dense passage retrieval` | fusion/retrieval статті за точними словами | RAG / IR статті за змістом | 1 спільний | **3 док.** |
| `graph neural networks for molecular property prediction` | ті самі molecular-GNN статті (CTAGE, MGNN, Dumpling GNN) | ті самі molecular-GNN статті | 3 спільних | 1 док. |
| `quantum error correction surface code` | точні «error correction» коди | «surface code» / fault-tolerance статті | 0 спільних | **4 док.** |

**Спостереження і загальне правило:**
- Коли BM25 і vector **збігаються** (GNN-запит: 3 спільних у топ-5) - гібрид додає
  мало, бо обидва методи вже згодні між собою.
- Коли вони **розходяться** (quantum-запит: 0 спільних) - гібрид найцінніший: RRF
  відновив 4 релевантні документи, яких жоден метод сам не мав у топ-5.
- **BM25** виграє на **точних/рідкісних термінах** (ідентифікатори, абревіатури,
  назви методів); **Vector** - на **семантиці/парафразі**; **Hybrid (RRF, k=60)**
  найробастніший на змішаних запитах.
- Правило: точний-термін/лексика → **BM25**; семантика/парафраз → **vector**;
  змішане/потрібна надійність → **hybrid через RRF**.

### 2. Розмір чанка: замалий vs завеликий

- **Замалий (10-15 слів)** - чанк без контексту: ембединг шумний, релевантний
  фрагмент дробиться, падає precision.
- **Завеликий (~500 слів)** - повертаємося до усереднення: один вектор «розмазує»
  локальні деталі, конкретна фраза тоне в загальному сенсі.

**Чи є оптимум?** Універсального немає - він **залежить від задачі** (довжина
документів, стиль запитів, контекстне вікно моделі). На практиці працює помірний
діапазон (десятки-сотні слів) із невеликим overlap; для нашого корпусу коротких
абстрактів навіть один вектор на документ часто достатній.

### 3. Неправильна метрика: euclidean (L2) для нормалізованих векторів

Нехай модель повертає **нормалізовані** вектори (`|a| = |b| = 1`), а індекс
помилково створено з евклідовою метрикою. Виведемо зв'язок:

```
L2(a, b)^2 = |a|^2 + |b|^2 - 2 (a · b) = 1 + 1 - 2 (a · b) = 2 - 2 cos(a, b)
```

Отже `L2^2 = 2 - 2·cos`: евклідова відстань **монотонно спадає** зі зростанням
косинуса. Тому **порядок сусідів (ранжування) тотожний косинусному** - топ
збігається. Але **самі значення** інші: cosine ∈ [-1, 1] (більше = ближче),
а L2 - відстань (менше = ближче). Тобто з L2 на нормалізованих векторах
ранжування правильне, проте збережені/повернені «дистанції» вже не косинусні
бали, і будь-які пороги/трешхолди, налаштовані під cosine, треба переобчислювати.

### 4. Ліміти free-tier і масштабування до 10 млн статей

Безкоштовний Starter-tier Pinecone обмежений (кількість/розмір індексів, обсяг
векторів, RPS). При переході з 10k на **10 млн** статей застосовуємо:

- **Шардинг / namespaces** - розбити корпус (за категорією/роком) на namespaces
  чи шарди, щоб обмежити простір пошуку й рознести навантаження.
- **Batched upserts + streaming ingestion** - заливати потоково великими
  батчами, а не разовим завантаженням; ідемпотентні id для дозаливки.
- **Зниження розмірності / квантизація** - PCA/matryoshka-ембединги, product
  quantization (PQ) або int8, щоб різко зменшити пам'ять і вартість.
- **Тюнінг ANN-індексу** - підбір параметрів HNSW (M, efSearch/efConstruction)
  або IVF-PQ під потрібний баланс recall/latency/memory.
- **Перехід на платний tier / горизонтальне масштабування** - більше подів/реплік
  під QPS і латентність, моніторинг та реіндексація під дрейф розподілу.

## Примітки

- Коментарі в коді - англійською; пояснювальний текст README - українською (вимога курсу).

  представницькій вибірці 8000 статей (Pinecone, SPECTER2). На іншій вибірці/машині
  конкретні статті відрізнятимуться, але закономірності (перетин методів, виграш RRF,
  обрізані fixed-чанки) відтворюються.
